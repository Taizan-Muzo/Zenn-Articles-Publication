# TL;DR

LLMのChain-of-Thought推論は、出力が「正しい」だけでなく、**実行器（executor）にとって実際に有用である**ことが重要だ。TraceLiftは、推論軌跡（reasoning trace）を単なる最終解答ではなく、実行器のパフォーマンス向上度を直接測定して報酬設計する新アプローチを提案する。実行器の改善度（uplift）を報酬の構成要素に組み込むことで、従来のLLM-as-Judgeでは検出できない微妙な推論品質の差を捉え、3つのモデルで一貫した改善を達成した。

```text
従来の報酬設計: 出力が正しいかどうか → 大部分のサンプルで飽和
TraceLiftの報酬: 出力が正しい ＋ 実行器が実際に改善するか → 品質差を識別
```

# 背景：推論品質評価のボトルネック

## 推論軌跡は「消費財」である

現在のLLM推論研究では、Chain-of-Thought（CoT）などの推論軌跡を生成品質の指標として扱うことが多い。しかし、TraceLiftの著者らは重要な視点を提示する——**推論軌跡は生成されるだけでなく、消費される中間生成物である**。

推論計画（planner）が生成した軌跡は、最終的に実行器（executor）——コードインタプリタや数学ソルバなど——に入力されて初めて価値を生む。この「プランナー・エグゼキュータ」の枠組みは、Reasoning with ExecutorsやReActなどで広く採用されている。

## LLM-as-Judgeの飽和問題

従来の推論品質評価では、LLMを判定器（judge）として用いるアプローチが主流だ。しかし、TraceLiftの分析が鮮やかにこの限界を暴く：

> **LLM-as-Judgeは95.5%のサンプルに最大スコア（1.0）を付与した。**

つまり、「正しい」推論軌跡間の微妙な品質差を、LLMのテキストベース評価ではほぼ区別できない。実行器にとって**本当に有用な推論**と、表面的には正しくても冗長な推論を弁別できないのだ。

# 手法詳解

## Executor-Grounded Rewardの定義

TraceLiftの中核は、実行器を評価の「接地（grounding）」として利用する報酬関数の設計にある。

$$R = 0.5 \cdot R_{exec} + 0.5 \cdot RM_\phi(P, R) \cdot u_{exec}(P, R)$$

この式は3つの構成要素を持つ：

**1) 実行器正解率 $R_{exec}$**: 実行器が推論軌跡を参照して正解を導けたかどうかのバイナリ評価。

**2) 推論品質スコア $RM_\phi(P, R)$**: ルーブリック（rubric）ベースのReasoning Reward Model（RM）。コード生成なら7次元、数学なら5次元で多角的に推論品質を評価する。従来の単一正解判定とは異なり、「冗長性」「ステップの妥当性」「効率性」なども考慮する。

**3) Uplift係数 $u_{exec}(P, R)$**: これがTraceLiftの最大の特徴だ。

$$u_{exec} = \text{clip}(\hat{p}(P, R) - \hat{p}_0(P), -1, 1)$$

ここで $\hat{p}(P, R)$ は推論軌跡R付きの実行成功率、$\hat{p}_0(P)$ は軌跡なしのベースライン成功率を表す。つまり、**推論軌跡があることで実行器の成功率がどれだけ上がったか**を測定する。

この係数の意義は深い。単に「推論が正しい」だけでなく、「推論が実行器に実際に利益をもたらしたか」を重み付けする。実行器が元々解ける問題に対しては $u_{exec} \approx 0$ となり、推論軌跡に高い品質スコアが割り当てられない。逆に、推論軌跡が決定的なヒントとなった場合には高い報酬が与えられる。

## TraceLift-Groupsデータセット

報酬モデルの訓練とプランナーの学習の両方を支えるデータセットが「TraceLift-Groups」だ。

- **6,000推論グループ**（コード3,000 + 数学3,000）
- 各グループには**1つの参照軌跡**と**平均6.28本の欠陥軌跡**
- 欠陥軌跡は**7種類の摂動（perturbation）**で生成：
  - 部分削除、順序シャッフル、冗長追加、ステップ置換、論理反転、ノイズ挿入、情報欠落

この設計の優れた点は、同一問題に対する品質グラデーションを持つ軌跡群を提供することだ。LLM-as-Judgeが「全部1.0」をつけてしまうような微細な差異を、実行器のupliftで浮き彫りにできる。

## Reasoning RMの訓練

Reasoning Reward Modelは、5つのルーブリック次元（数学の場合）または7次元（コードの場合）で推論品質を評価する。

- 損失関数は**次元別スコア + 総合スコア + ペアワイズランキング**の3つを組み合わせた複合損失
- **ペアワイズ精度99.15%**を達成
- LLM-as-Judgeの飽和とは対照的に、品質差を高精度で弁別可能

## プランナーの訓練

プランナー（reasoning planner）の訓練にはGRPO（Group Relative Policy Optimization）を用いる。

- Executor-Grounded Rewardを直接最適化
- uplitf項によるフィルタリング効果で、実行器にとって無意味な「正しいだけの推論」を抑制
- 訓練中は推論長がむしろ**短縮**される傾向を示す（後述）

# 実験結果

![fig2](/images/tracelift-executor-grounded-reasoning-rewards/fig2.png)

*図2: 3つのモデル（Qwen2.5-7B, Llama3.1-8B, Qwen3-4B）でのメイン結果。青が実行器単体、オレンジがTraceLift。全モデルで一貫した改善。*

## メイン結果

| モデル | ドメイン | Exec-only | TraceLift | 改善幅 |
|--------|--------|-----------|-----------|--------|
| Qwen2.5-7B | Code | 73.4 | **76.0** | +2.61pp |
| Qwen2.5-7B | Math | 74.2 | **78.7** | +4.51pp |
| Llama3.1-8B | Code | 72.7 | **74.7** | +1.96pp |
| Llama3.1-8B | Math | 74.0 | **75.7** | +1.69pp |
| Qwen3-4B | Code | 69.2 | **71.6** | +2.44pp |
| Qwen3-4B | Math | 71.7 | **72.5** | +0.84pp |

**3つの異なるモデルで一貫して改善**を確認。特にQwen2.5-7Bでの数学ドメイン+4.51ppの改善が顕著で、最大のモデルサイズではないにもかかわらず最良のパフォーマンスを示した。

![fig3](/images/tracelift-executor-grounded-reasoning-rewards/fig3.png)

*図3: Qwen2.5-7B コード生成でのアブレーション結果。Uplift項の除去が最も大きな性能低下をもたらす。*

## アブレーション分析

Qwen2.5-7Bでのアブレーションが、各構成要素の重要性を明確に示している：

| 設定 | マイクロ平均 | TraceLiftからの差 |
|------|-------------|-------------------|
| **TraceLift（完全版）** | **76.3** | — |
| RM-upliftのみ（$R_{exec}$除去） | 75.3 | -1.01pp |
| No-uplift（$u_{exec}$除去） | 73.0 | **-3.34pp** |
| LLM-as-Judge | 73.0 | **-3.34pp** |
| Exec-only（訓練なし） | 71.9 | -4.44pp |

最も注目すべきは**No-upliftとLLM-as-Judgeが同スコア（73.0）**である点だ。uplift係数を取り除くと、TraceLiftのRMは本質的にLLM-as-Judgeと等価な性能に退化する。この結果は、**品質差の識別力がuplift接地から来ている**ことを決定的に示している。

## 推論長の分析

![fig4](/images/tracelift-executor-grounded-reasoning-rewards/fig4.png)

*図4: 訓練ウィンドウごとの推論長（上）とuplift係数（下）の推移。推論長は短縮傾向を示す。*

推論長の分析は意外な発見を含んでいる：

| モデル | Exec-onlyの推論長 | TraceLiftの推論長 |
|--------|-------------------|-------------------|
| Qwen2.5-7B | 166.0 | **136.3** |
| Llama3.1-8B | 153.0 | **143.5** |
| Qwen3-4B | 144.3 | **130.1** |

**TraceLiftの推論軌跡はExec-onlyよりも短い**。これは直感に反するかもしれない——より良い推論はより長くなるのではないか？しかし、upliftに基づく訓練は「実行器にとって冗長なステップ」を淘汰する方向に働く。余分な推論ステップが実行器の成功率に寄与しない場合、そのステップは報酬を低下させるため、効率的で本質的な推論が選ばれるのだ。

これは推論の「効率性」が品質の重要な次元であることを示唆している。

# 考察

## 「正しさ」と「有用性」の分離

TraceLiftの最大の貢献は、推論軌跡の評価において**正しさ（correctness）と有用性（utility）を明示的に分離した**点にある。

従来のアプローチでは、推論が論理的に正しいことと、それが下流タスクに有用であることが暗黙に同一視されていた。しかし、実験が示すように、LLM-as-Judgeの「正しさ」評価は飽和し、品質のグラデーションを捉えられない。TraceLiftは、実行器という具体的な「消費者」を評価のアンカーにすることで、この問題を解決した。

## スケールの普遍性

3つの異なるアーキテクチャ（Qwen, Llama, Qwen3）で一貫した改善が見られたことは、このアプローチの普遍性を示唆している。特に、**Qwen3-4B（最小モデル）でも+2.44ppのコード改善**を達成した点は、小規模モデルでの効率的な推論訓練への応用可能性を示している。

## データセット設計の巧妙さ

TraceLift-Groupsの「7種類の摂動 × 参照軌跡」の設計は、品質グラデーションを系統的に生成する枠組みとして優れている。ルーブリックベースのRM訓練に必要な「どの推論がどれくらい良いか」のシグナルを、実行器のupliftを通じて獲得できる点が肝要だ。

## 限界と展望

- **実行器への依存**: upliftの計算には凍結された実行器が必要で、実行器自体の品質がボトルネックになりうる
- **2段階パイプライン**: プランナーと実行器の分離はシンプルだが、エンドツーエンドの推論では追加のオーバーヘッドとなる
- **ドメイン**: 現在はコード生成と数学推論に限定されており、より開放的な推論タスクへの拡張が今後の課題

# 関連研究

- **Reasoning with Executors**: プランナー・エグゼキュータ枠組みの先駆的研究。TraceLiftはこの枠組みの報酬設計に特化した改善
- **GRPO / RLVR**: プランナー訓練に用いた強化学習アルゴリズム。近傍のRLVR研究（CIPO, SCRL等）とも文脈を共有
- **DPO / On-Policy Methods**: 比較対照となる報酬設計アプローチ。TraceLiftの報酬関数はGRPO内で利用される点でDPOとは異なる
- **LLM-as-Judge研究**: 推論品質評価の標準的アプローチだが、TraceLiftはその限界を実証的に示した

# まとめ

TraceLiftは、「正しい推論」と「有用な推論」の区別を**実行器のパフォーマンス改善度（uplift）**という客観的指標で定式化した。LLM-as-Judgeの飽和問題を実データで実証し、uplift接地による報酬設計がその問題を実用的に解決することを3つのモデルで示した。

特に興味深いのは、upliftベースの訓練が推論を**短く、効率的に**することだ。これは「良い推論＝論理的に完全な推論」という暗黙の前提を問い直す結果であり、推論の効率性が品質の独立した次元であることを示している。

推論計画の品質評価に、その消費者の反応を直接組み込む——シンプルだが強力なアイデアだ。
