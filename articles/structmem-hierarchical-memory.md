# 構造化記憶でAgentの「長期記憶」を実現するStructMem

## TL;DR

- 長期対話Agentの記憶には**事実の断片**だけでなく**イベント間の関係性**が必要
- StructMemは「イベント中心」の階層化記憶で、**FlatとGraphの二項対立を打破**
- LoCoMoベンチマークで**Overall 76.82%**を達成（既存SOTA +1.04）
- Token消費はGraph Memory比で**18.5倍の効率**、API呼び出しは**50倍削減**
- ACL 2026主会採択。浙大・アリババ知識グラフ連携ラボの研究成果

## なぜこの論文が重要か

LLM Agentが実用的なアプリケーションで動くためには、一度きりの対話では済まない。何週間、何ヶ月にもわたる長期の会話をまたいで、ユーザーの文脈を維持し、過去の出来事を正しく引き出せる「記憶」が不可欠だ。

ところが、現状の記憶システムには構造的なジレンマがある。

**Flat Memory**（LangMemやMem0など）は効率が高い。会話から事実を抽出してリストに並べ、ベクトル検索で引き出す。シンプルで堅牢だが、事実と事実の「つながり」が消えてしまう。「AさんとBさんが一緒にイベントに行った」という関係性がバラバラの条目になってしまえば、多段推論や時系列推論が成立しない。

一方、**Graph Memory**はエンティティとリレーションの三元組で構造を保持する。理論上は強力だが、実際にはLLMが構築したグラフの品質が低く、構築コストは**イベントごとに4回のカスケードLLM呼び出し**。重複除去の計算量も二次関数的に増大する。Graph MemoryのToken消費はFlatの3〜18倍に達し、実用性に疑問符がつく。

StructMemは、このトレードオフを「イベント中心の階層化」という新しいアプローチで解こうとした論文だ。

## StructMemの設計思想

StructMemの根底にあるのは、ひとつの問いだ。

> 記憶の構造を保持しつつ、構築コストをFlat Memory並みに抑えられるか？

答えは「階層化」にある。StructMemは記憶を2層に分ける。

### 第1層：イベントレベルの双方向抽出

会話の各セグメントから、**事実条目**と**関係条目**を同時に抽出する。

- 事実条目：「Aさんがレストランでパスタを注文した」（何が起きたか）
- 関係条目：「AさんはBさんに料理の好みを聞いた」（誰と誰の間にどんな動機/因果があるか）

Flat Memoryは前者だけを抽出する。Graph Memoryは後者を三元組に変換しようとするが、粒度が粗い。StructMemは両方を**自然言語のまま**、しかも**タイムスタンプ付き**で保持する。これが「時間アンカー付き双方向プロトコル」だ。

### 第2層：周期的事象統合

ここがStructMemの核になる。

時間バッファにイベントが蓄積されると、**一定期間（1時間）ごと**に統合フェーズが発動する。手順は以下の通り：

1. 埋め込み類似度で、類似する過去のイベントを検索（種子数 K=15）
2. セマンティッククラスタリングでイベントをグループ化
3. **関係仮説の合成** — 各クラスタから、イベントをまたぐ新たな関係記述を生成する

キーポイントは「有損テキスト要約」でも「剛直なグラフマージ」でもなく、**新たな関係仮説を合成**する点にある。これにより、「AさんとBさんが昨年一緒にプライドフェスティバルに行った」といった、どの単一記憶条目にも存在しない**上位の抽象**が生まれる。

## 実験結果

### LoCoMoベンチマーク

LoCoMoは長期対話記憶の評価用ベンチマークで、平均**588ターン**、**16,618トークン**という非常に長い対話が含まれる。全部で1,540問（Single-hop 841問、Multi-hop 282問、Temporal 321問、Open-domain 96問）。

全手法でバックボーンモデルを**gpt-4o-mini**に統一し、LLM-as-a-Judgeで評価している。

**全体精度で76.82%**を記録し、次点のMemobase（75.78%）を+1.04上回った。特に注目すべきは、Structured Memoryの中ではZepやMemobaseと同等以上の精度を、**圧倒的に少ないリソースで**達成している点だ。

### 効率の比較

![効率性の比較（StructMem基準の相対値）](/images/structmem-hierarchical-memory/fig2_efficiency_comparison.png)

この図がStructMemの最大の武器だ。全指標でStructMemを「1.0x」として他手法の相対値を示している。

- **Build Token**: StructMem 1.501M。最も近いMemoryOS（1.889M）に対して**20.5%削減**。Mem0^g（33.512M）とは**22.3倍**の差
- **Query Token**: StructMem 0.436M。全手法中最低
- **API Calls**: StructMem 1,056回。Mem0^g（53,514回）の**50.7分の1**

Graph Memoryの問題がここに如実に現れている。構造を構築するためのLLM呼び出しが洪水のように増え、しかも重複除去のコストが二次関数的に膨らむ。StructMemはイベントバッファとバッチ処理でこのコストを「償却」する。イベントごとの操作を周期的なバッチにまとめることで、Graph Memoryの逐次カスケード構築を回避している。

### 消融実験：各コンポーネントの寄与

![消融実験：パラダイムの影響](/images/structmem-hierarchical-memory/fig3_ablation_paradigm.png)

消融実験から2つの重要な発見がある。

第一に、**Graph MemoryはTemporal推論でFlat Memoryより劣る**（78.50%→76.64%、-1.86pp）。Graph構築のノイズが時系列推論を阻害するということだ。構造があっても品質が低ければ、むしろ有害になる。

第二に、StructMemの完全版が**Temporal 81.62%**を記録し、Flat単体（78.50%）から+3.12ppの大幅改善。これは単にイベント構造を足しただけでは説明できない。Cross-Event統合が、単一イベントの記憶では決して得られない**跨時間境界の因果関係**を再構築している証拠だ。

検索条目数の分析（原論文図3c）でも、Flat検索は60条目でプラトーに達する。事実を追加しても推論能力は上がらない。ボトルネックは「知識のカバレッジ」ではなく「知識の構造化」にある。

### Cross-Judge堅牢性と幻覚率

![複数Judgeモデルでの堅牢性と幻覚率](/images/structmem-hierarchical-memory/fig4_robustness_hallucination.png)

3つの異なるJudgeモデル（GPT-4o-mini、Qwen2.5-32B、DeepSeek-V3.2）で評価し、全JudgeでStructMemが最高のOverallスコアを記録した。Judge間のFleiss' κは**0.8341**で、評価の一貫性も担保されている。

幻覚率の分析も興味深い。

- イベント抽出の幻覚率は**平均2.36%**。LLMベースの抽出としては非常に低い水準だ
- Cross-Event統合の幻覚率は、時間アンカー制約ありで**0.61〜3.63%**。制約なしでは**7.45〜20.00%**に激増する

この対比が示唆するのは、StructMemの高忠実度が「制約付き合成」メカニズムに大きく依存している点だ。LLMに自由に関連付けさせるだけでは幻覚が爆発する。時間的・具体的なアンカーで合成プロセスをガードレールで囲うことが、実用記憶システムの要件になる。

## 考察：記憶システムのデザインパターン

StructMemの設計には、エージェントシステムのアーキテクチャ設計一般に通じる教訓がある。

**「毎回フルの構築コストを払う必要はない」**

Graph Memoryの失敗は、イベントごとに構造を再構築しようとした点にある。StructMemの周期統合は、データベースのバッチインデックス再構築や、Incremental View Maintenanceに似ている。変化をバッファに溜め込み、定期的に一括処理する。これは分散システムにおける定石だ。

**「構造の粒度を間違えると、構造がないより悪い」**

消融実験でGraph Memory < Flat Memoryとなったタスクがある事実は重い。リレーション抽出の品質がまだ高くない現在においては、剛直な三元組構造はノイズを増幅しかねない。StructMemの「自然言語のまま関係を保持し、統合で抽象を合成する」という柔軟なアプローチは、現実的な妥当点だと言える。

**「制約はクリエイティビティの敵ではない、品質の味方だ」**

制約なし合成の幻覚率が制約ありの5〜12倍になる結果は、LLMの創造性を「正しい方向」に誘導するガードレールの重要性を示している。System Promptの設計において、何を許容し何を禁止するかのバランスは、応用領域ごとに慎重に設計すべきだ。

## 関連研究の位置づけ

記憶システムの研究は、近年Agentコミュニティで活発だ。

- **MemGPT / MemoryOS**：OSの仮想メモリ管理に着想を得た階層型記憶。StructMemと階層化の方向性は共通するが、イベント中心の構造化はStructMem独自
- **SCIMO**：Graph Memoryの代表例。高い表現力を持つが、構築コストの壁に直面
- **A-Mem / Mem0**：Flat Memoryの主流実装。効率と堅牢性で優れるが、関係推論が弱点
- **HiGMem / H-MEM**：LLM誘導型の階層記憶。構造構築にLLMの推論力を活用する点でStructMemと共通
- **KnowRL**（本シリーズで過去に取り上げた論文）：知識の最小十分性に焦点を当てたRLVR研究。StructMemの「最小限のCross-Event統合で最大の推論効果」という設計哲学に通じる

StructMemの独自の寄与は、「FlatとGraphの中間」に位置する**イベント中心階層化**という新しいデザインポイントを提示した点にある。

## まとめ

StructMemは、長期対話Agentの記憶設計に対して明確な回答を提示した。

- **Flat Memoryの効率**と**Graph Memoryの構造化**を、イベント中心の階層化で統合
- **双方向抽出**で事実と関係を同時に捕捉し、**周期統合**で跨イベント抽象を合成
- LoCoMoで最高精度を、既存構造型記憶の**18.5倍の効率**で達成
- 制約付き合成の幻覚率は**<4%**、制約なしでは5〜12倍に悪化することを実証

ACL 2026主会への採択は、このアプローチの妥当性を示している。実用Agentの記憶層として、非常に検討に値する設計だ。
