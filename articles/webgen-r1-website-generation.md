# TL;DR

- **WebGen-R1**は7BパラメータのLLMをRLで訓練し、機能的かつ美しいマルチページWebサイトを**エンドツーエンド**で生成するフレームワーク（ICLR 2026）
- **スキャフォールド駆動構造化生成**でアクション空間を制約し、プロジェクトの構造的完全性を保証
- **3段階カスケード型マルチモーダル報酬**（構造検証 → 実行ベース機能評価 → VLMによる美学的監督）を設計
- ベースモデルからVRR 30.56% → **95.89%**、FSR 1.59% → **29.21%** に劇的改善
- DeepSeek-R1（671B）にほぼ匹敵するFSRを達成しつつ、VRRと美学スコアで**大幅に上回る**

# なぜこの論文が重要か

LLMのコード生成能力は飛躍的に向上してきたが、その大部分は**関数レベル**（単一ファイルの関数やモジュール）に限定されている。現実のソフトウェア開発では、複数ファイルの依存関係、動的ルーティング、状態管理、認証フロー、クロスページナビゲーションなど、はるかに複雑な要件を満たす必要がある。

Webサイト生成という文脈では、既存のアプローチは大きく2つの課題を抱えていた。

1. **単一ページへの限定**: 静的な単一HTMLページの生成に留まり、現代的なWebアプリの複雑性を無視している
2. **エージェントフレームワークへの依存**: 複数の特化サブエージェントを連携させるマルチターン実行が必要で、膨大なトークン消費・高レイテンシ・脆弱な統合が生じる

小規模オープンモデルをRLでエンドツーエンド訓練する方向性は有望だが、「主観的な美しさ」「クロスページの相互作用」「機能的正确性」を統合的に評価する報酬設計という**致命的なボトルネック**が存在した。WebGen-R1は、このボトルネックに真正面から取り組んだ。

# 手法の全体像

WebGen-R1は、香港科技大学・Alibaba Tongyi Lab・ETRI・Ant Groupの共同研究で、Qwen2.5-Coder-7B-Instructをベースモデルに用いる。フレームワークは4つの構成要素から成る。

![Cascaded Multimodal Reward Architecture](/images/webgen-r1-website-generation/fig1_reward_architecture.png)

## 1. スキャフォールド駆動構造化生成

プロジェクト全体をゼロから生成するのではなく、**事前検証済みのReactテンプレート**を「スキャフォールド（足場）」として固定する。テンプレートにはビルド設定、ルーティングスケルトン、サーバーサイドロジック、ディレクトリ構造の不変部分が含まれる。

LLMのポリシーは固定スキャフォールドの**外部にある変数コンポーネント**（ページ、コマンド、スタイルなど）のみを生成する。これによりプロジェクトの基本的な構造は**先験的に保証**され、巨大なアクション空間の探索が現実的な範囲に収まる。

## 2. 階層型検証・レンダリングパイプライン

生成されたプロジェクトは、コストのかかる報酬モデリング前に**厳密なフィルタリング**を経る。

**Phase I: 静的コンプライアンス検証** — 構造制約、ファイル要件、コマンド要件、コンテンツルール（`package.json` の妥当性など）を検証。不合格なら早期打ち切り（報酬 = 0）。

**Phase II: 自動ビルド・レンダリング** — 依存解決 → バンドル（Vite） → ローカルサーブ起動 → ヘッドレスブラウザでレンダリング。各ルートのスクリーンショット、ランタイムログ、コンソールログを出力。

## 3. カスケード型マルチモーダル報酬

この論文の最も重要な貢献だ。

### 階層型報酬の全体構造

$$R(y) = \begin{cases} \psi_{static}(W_{gen}) & \text{if } I_{static} = 0 \\ \psi_{build}(\Lambda_{runtime}) & \text{if build fails} \\ R_{dense}(y, o, x) & \text{otherwise} \end{cases}$$

静的検証やビルド失敗の場合は報酬を0にし、高価なVLM推論を**不必要な候補に浪費しない**。

### 密報酬の3つの構成要素

正常にレンダリングできた場合、以下の重み付き線形結合を用いる：

$$R_{dense} = s_{vis} + \gamma \cdot s_{func} + \lambda \cdot s_{cot}$$

- **美学知覚スコア** ($s_{vis}$): VLMがスクリーンショットを評価。レイアウト調和、タイポグラフィ階層、機能的視覚的質、ユーザー意図とのスタイル一致を [0, 5] のスケールで採点
- **機能完全性スコア** ($s_{func}$): ランタイム・コンソールエラーが0なら1、それ以外は0（スパースバイナリ信号）
- **推論フォーマットスコア** ($s_{cot}$): Chain-of-Thought推論が `thinking` タグで囲まれていれば1（バイナリフォーマット報酬）

$s_{vis}$ が密な連続信号であるのに対し、$s_{func}$ と $s_{cot}$ はスパースなバイナリ信号だ。最適な重みパラメータは **$\gamma = 0.1, \lambda = 0.1$**。スパース構造報酬を過重すると視覚的忠実性が低下し、逆に視覚報酬だけに頼ると生成ロジックが弱まるというトレードオフが存在する。

## 4. GRPOによる方策最適化

PPOではなく **GRPO（Group Relative Policy Optimization）** を採用。同じプロンプトからG個の出力をサンプリングし、グループ内の相対的優位度で方策を更新する。

Webサイト生成では、微細な構文エラーがレンダリング失敗を引き起こし、報酬を高値からゼロに急落させるため報酬の分散が極めて高い。GRPOのグループ正規化は、この**高分散・高ボラティリティ**な報酬信号に対する最適化の安定性を向上させる。実験では **G = 16** が最適だった。

# 実験結果

![Model Comparison on WebGen-Bench](/images/webgen-r1-website-generation/fig2_model_comparison.png)

## WebGen-Benchでの比較

WebGen-Bench（101タスク）での結果は以下の通り。

| モデル | パラメータ数 | FSR (%) | AAS | VRR (%) |
|--------|-------------|---------|-----|---------|
| **WebGen-R1-7B** | **7B** | **29.21** | **3.94** | **95.89** |
| Qwen2.5-Coder-7B-Instruct | 7B | 1.59 | 2.73 | 30.56 |
| Qwen2.5-72B-Instruct | 72B | 2.54 | 3.14 | 8.86 |
| Qwen3-32B | 32B | 18.69 | 3.39 | 59.42 |
| DeepSeek-R1 | 671B | 30.25 | 3.67 | 42.86 |
| GPT-5 | - | 46.53 | 3.34 | 90.43 |
| Claude-3.7-Sonnet | - | 57.72 | 3.90 | 84.00 |
| Claude-Sonnet-4 | - | 46.13 | 3.86 | 86.05 |

WebGen-R1の注目点：

- **VRR 95.89%** — 全モデル中最高。671BのDeepSeek-R1（42.86%）を53pp上回る
- **AAS 3.94** — 全モデル中最高。Claude-3.7-Sonnet（3.90）を凌駕
- **FSR 29.21%** — ベースモデルから+27.62ppの改善。7つのオープンソースモデル中6つ（最大72Bまで）を一貫して上回り、DeepSeek-R1（30.25%）にほぼ匹敵

## 訓練パイプラインの貢献度

![Training Pipeline: SFT + RL](/images/webgen-r1-website-generation/fig3_training_pipeline.png)

SFT（600インスタンス、GPT-4.1蒸留）単独でも大幅な改善が見られるが、RLの追加がさらなるブーストをもたらす。

| 段階 | FSR (%) | VRR (%) | AAS |
|------|---------|---------|-----|
| Base | 1.59 | 30.56 | 2.73 |
| SFT | 20.08 | 30.69 | 3.52 |
| RL only | 18.23 | 26.82 | 2.96 |
| **SFT + RL** | **29.21** | **95.89** | **3.94** |

RL only（R1-Zero）ではVRRがベースより低下するという興味深い結果が見られた。SFTが構造的・意味的事前知識を提供し、RLが報酬ガイド下での探索を改善するという**相補的な役割分担**が確認された。

## グループサイズと報酬重みの分析

![Group Size & Reward Weights](/images/webgen-r1-website-generation/fig4_group_size_reward_weights.png)

**グループサイズG**の影響：G = 2 → G = 32 でFSR 22.39% → 34.79% と一貫した改善。より大きなグループが探索の多様性を高め、高品質なWebサイトを発見する確率を上げる。ただし論文では計算コストの観点からG = 16を採用。

**報酬重みの感度分析**：$\gamma$（機能完全性重み）を大きくするとFSRが低下する傾向があり、$\lambda$（推論フォーマット重み）を中程度に設定するとAASとVRRで安定した改善が得られる。最適設定 $\gamma = 0.1, \lambda = 0.1$ でFSR 29.21%を達成。

## 汎化性能

WebGen-Benchとは異なるドメイン・プロンプト分布を持つ **WebDev Arena**（119タスク）でも評価。FSRは標準テストケースがないため未測定だが、AASではGPT-5、Claude-Sonnet-4、DeepSeek-R1、Qwen3-32Bなどの強力なベースラインを一貫して上回った。これはRL訓練を通じて**アーキテクチャレベル・スタイルレベルの抽象化**を獲得したことを示唆している。

## 人間アライメント

3人の経験豊富なフロントエンド開発者に101サイトの機能性と視覚的魅力度を評価させた結果、報酬モデルと人間評価の間に**Pearson r = 0.762、Spearman ρ = 0.734** の強い相関が確認された。

# 考察

WebGen-R1の結果から、いくつか重要な示唆が得られる。

**機能的正しさはスケーリングだけでは解決しない**: Claude-3.7-Sonnet（FSR 57.72%）やGPT-5（FSR 46.53%）が最も高いFSRを示したが、美学スコアやレンダリング率ではWebGen-R1-7Bに劣る。デザイン美学はモデルスケールで比較的スケールしやすいが（Qwen3-32BのAAS 3.39など）、機能的正しさには表層的なレイアウト生成を超えた正確なロジックと相互作用処理が必要だ。

**スキャフォールドの威力**: 72BのQwen2.5-72B-InstructがVRR 8.86%に留まるのに対し、WebGen-R1-7Bが95.89%を達成している。パラメータ数の差は64倍だが、スキャフォールドによる構造的保証がこのギャップを圧倒的に凌駕している。

**報酬設計の工夫がRLVRの実用性を左右する**: スパースなバイナリ信号（$s_{func}$, $s_{cot}$）と密な連続信号（$s_{vis}$）を適切にバランスさせ、階層的な早期打ち切りで計算効率を担保する設計は、Webサイト生成に限らず「主観的品質」と「客観的正しさ」の両方を最適化するタスク全般に応用可能な洞察だ。

**限界**: はっきりしているのは、Claude-3.7-SonnetなどのトッププロプライエタリモデルとはFSRでまだ28pp以上の開きがあること。またWebDev ArenaのOOD環境では、指示の詳細度が低いタスクで最新のデザイントレンドを捕捉できず、ベースモデルの世界知識に依存する傾向が見られた。

# 関連研究との位置づけ

| 研究 | 接近法 | 規模 | 特色 |
|------|--------|------|------|
| WebGen-Bench | 評価ベンチマーク | - | WebGen-R1の評価基盤 |
| WebVoyager / MetaGPT | マルチエージェント | 依存 | 複雑なオーケストレーションが必要 |
| WebGen-LM | SFT（Bolt.diy） | - | 特定エージェントに結合、エンドツーエンドで非機能 |
| DeepSeek-R1 | RL（R1-Zero） | 671B | 推論タスク向け、Webサイト生成は非最適 |
| **WebGen-R1** | **SFT + RL** | **7B** | **スキャフォールド + カスケード報酬で初のエンドツーエンドRL** |

WebGen-R1の最大の差別化要因は、「マルチファイルの構造的依存、実行妥当性、機能的正しさ、視覚的品質を**統合的なRLフレームワーク内で同時に最適化**した点」にある。

# まとめ

WebGen-R1は、小規模オープンモデルをプロジェクトレベルのWebアプリ生成に適用可能にするRLフレームワークとして、以下の実証を行った。

1. **スキャフォールド駆動生成**がアクション空間を制約し、構造的完全性を維持
2. **カスケード型マルチモーダル報酬**が機能性・美学・実行妥当性を統合的に評価
3. 7Bモデルが72Bオープンソースモデルを一貫して上回り、671BのDeepSeek-R1にほぼ匹敵
4. VRR 95.89%（全モデル最高）、AAS 3.94（全モデル最高）を達成

「関数レベルのコード生成からプロジェクトレベルのWebアプリ生成へのスケーリング」という長年の課題に対し、RL + スキャフォールド + マルチモーダル報酬という方向性が有効な道筋であることを示した重要な論文だ。
