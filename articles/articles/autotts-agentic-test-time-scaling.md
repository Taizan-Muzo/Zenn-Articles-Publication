## TL;DR

- AutoTTSは、LLMのテスト時スケーリング（TTS）戦略を**環境駆動で自動発見**するフレームワーク
- 幅（BRANCH/PRUNE）と深さ（CONTINUE）の5つのアクションで推論空間を探索
- **βパラメタライゼーション**で探索空間を1次元に圧縮、過学習を回避
- 発見されたCMCコントローラは**SC@64比で約69.5%のトークン削減**を達成しつつ精度を維持
- 探索の全コストは**わずか$39.9・160分**、LLM呼び出しゼロのオフラインリプレイ環境

## なぜこの論文が重要か

テスト時スケーリング（Test-Time Scaling, TTS）は、推論時に計算量を追加投入することでLLMの性能を引き上げるアプローチとして、ここ1年で急速に広まった。Self-Consistency（SC@64）をはじめ、ASC、ESC、Parallel-Probeなど、様々な手作りヒューリスティクスが提案されてきた。

しかし、これらの戦略はすべて**研究者の直感に依存**している。幅と深さの組み合わせ、枝刈りのタイミング、停止条件……膨大な設計空間のうち、人間が手作業で探索できるのはごく一部に過ぎない。

> "Test-time scaling strategies are largely hand-crafted: researchers manually design reasoning patterns and tune heuristics by intuition, leaving much of the computation-allocation space unexplored."

AutoTTSは、このパラダイムを根底から覆す。**戦略そのものではなく、戦略が発見される環境を設計する**という視点転換だ。そして、その発見を支えるのがClaude Codeのようなコーディングエージェントである——LLM自身が、LLMのための推論戦略を書き上げる。

## TTSをアルゴリズム探索問題として定式化

### 状態・アクション・コスト

AutoTTSはTTSを**コントローラー合成問題**として捉える。状態はタプル $s_t = (q, m_t, I_t, \ell_t, Z_t, \Omega_t)$ で表され、コントローラーは各ステップで以下の5つのアクションから選択する：

| アクション | 意味 |
|-----------|------|
| **BRANCH** | 新たな推論枝を生成（幅の拡張） |
| **CONTINUE(i)** | 枝$i$を1インターバル延長（深さの拡張） |
| **PROBE(i)** | 枝$i$の進捗信号$\omega$を観測（生成なし） |
| **PRUNE(i)** | 枝$i$を非活性化（過去情報は保持） |
| **ANSWER** | 推論を終了し集約ルールで最終回答を出力 |

コスト関数は $Cost(s_t) = \sum_i \ell_{t,i} + \kappa_{probe}|\Omega_t|$ で定義される。プローブが生成と比べて十分安価な場合、$\kappa_{probe} = 0$ として扱う。

最適化目的は、正解率とコストのトレードオフを制御するスカラー$\gamma$を用いて：

$$\max_{(\pi, \beta)} \mathbb{E}_{(q,y) \sim D, \tau \sim P_{\pi,\beta}(\cdot|q)} [\mathbf{1}\{\hat{y}_{\pi,\beta}(\tau) = y\} - \gamma \cdot C(\tau)]$$

### 既存手法の配置図

![Existing TTS methods in width-depth space](/images/autotts-agentic-test-time-scaling/fig2.png)

既存手法はいずれも幅・深さ空間の限定的な領域しか探索していない：
- **SC@64**: 固定の全予算コーナー（幅64・最大深さ）
- **ASC/ESC**: 幅方向のみ適応、深さは最大固定
- **Parallel-Probe**: 幅からスタートし漸進的に枝刈りしながら深化

AutoTTSが目指すのは、この空間全体を自動探索することだ。

## βパラメタライゼーション：探索空間の1次元化

### 問題：過学習と探索の破綻

事前実験で、探索エージェントは**最大10個のハイパーパラメータ**を持つコントローラーを提案した。しかし発見ラウンドはわずか5回。10次元空間を5回の反復で探索すると、エージェントは鋭い閾値（例：過度に攻撃的な枝刈り）に収束し、探索セットに過学習してしまう。

### 解法：単一スカラー + 単調写像

βパラメタライゼーションは、コントローラーに2つの制約を課す：

1. **単一ハイパーパラメータ**: 各コントローラーはスカラー$\beta$のみを公開する
2. **単調性**: $\beta$から内部パラメータへの写像は単調——$\beta$が大きいほどトークン予算が大きい

これにより探索空間は**1次元のスイープ**に圧縮され、探索セット特有の鋭い閾値の発見が防止される。写像関数自体はコーディングエージェント（Claude Code）が生成する。

## オフラインリプレイ環境

### ゼロLLM呼び出しの評価

AutoTTSの鍵は、発見環境の構築にある。Parallel-Probeのプロトコルに従い、探索フェーズ開始前に全データを収集する：

- 各問題につき**N=128本の独立推論軌跡**を事前収集
- サンプリング温度 **0.7**、プローブ間隔 **Δ=500トークン**
- 軌跡を500トークン単位の固定長セグメントに分割して保存

コントローラーの評価は**オフラインリプレイ**で行う。PROBEアクションは保存済みの信号$\omega_{i,k}$を検索するだけで、**新しいLLM呼び出しは一切不要**。$\beta$全体のスイープが追加コストなしで実行可能だ。

分散を低減するため、各コントローラーの評価は128本の軌跡プールからランダムにサブセットを抽出する試行を**64回独立に**行い、結果を平均化する。

### 発見ループ

1. エージェント（Claude Code）が蓄積履歴$\mathcal{H}$を読み込み、コントローラーのコードを直接編集して改善案を提案
2. 提案されたコントローラーを$\mathcal{Q}_{search}$（AIME24）で評価
3. 結果（正解率・コスト、および**実行トレース**全体）を$\mathcal{H}$に追記
4. これを5ラウンド繰り返し、最良の$(\pi, \beta)$を選択

重要なのは、履歴$\mathcal{H}$がスカラーメトリクスだけでなく**詳細な実行トレース**を含む点だ。これによりエージェントは「なぜその戦略が失敗したのか」を診断し、的確な改善を提案できる。

## 発見されたコントローラー：CMC

5ラウンドの発見を経て選ばれたのが、**CMC（Confidence Momentum Controller）**だ。

### 4つのコアメカニズム

**1. トレンドベース停止**
指数移動平均（EMA）によるプール信頼度のモメンタムを監視し、信頼度が高水準**かつ**トレンドが非負のときのみ停止を許可する。一時的な信頼度スパイクによる早期停止を防ぐ。

**2. 結合型幅・深さ制御**
信頼度の上昇が大きいほど新規枝の生成を抑制し、停滞や回帰が起きると幅を拡大する。

**3. アライメント認識型深さ配分**
プールの勝者回答と一致する枝には追加のプローブステップを配分し、合意が形成されつつある枝に計算を集中させる。

**4. 保守的な枝放棄**
枝の放棄は持続的な逸脱後にのみ実行され、常に**最低2本のアクティブ枝**を維持する。

### βスケジュール

$\beta \in [0, 1]$から内部パラメータへの単調写像は以下の通り：

```python
def _schedule(self, beta: float) -> dict:
    b = max(0.0, min(1.0, float(beta)))
    n_init           = max(2, round(2  + 6  * b))
    max_branch_use   = min(64, round(4 + 60 * b))
    warm_up          = max(2, round(2  + 8  * b))
    abandon_patience = max(3, round(3  + 9  * b))
    T_ema            = max(2, round(2  + 6  * b))
    ema_alpha        = 0.70 - 0.40 * b
    conf_thresh      = 0.85 + 0.12 * b
    delta_slack      = 0.04 - 0.03 * b
```

### 3層の枝分類

| 層 | 条件 | プローブ倍率 |
|---|------|------------|
| Aligned | 最新回答 == プール勝者 | `burst_aligned`（高β時は2） |
| Neutral | 勝者未確定、または初回不一致 | 1 |
| Deviant | 1ラウンド以上不一致 | 1（`abandon_patience`後に放棄） |

### EMAモメンタムゲート

```python
gate_fires = (
    gate_eligible
    and ema_conf >= self.conf_thresh      # 水準要件
    and ema_delta >= -self.delta_slack    # 非悪化モメンタム
)
```

停止判断に「方向」を導入することで、CMCは単純な閾値ベースの手法よりはるかにロバストな振る舞いを実現している。

## 実験結果

![Accuracy-cost scaling curves](/images/autotts-agentic-test-time-scaling/fig3.png)

### スケーリングカーブの比較

AIME25におけるQwen3-1.7Bのスケーリングカーブを見ると、CMCコントローラーが手作りベースライン全体のParetoフロンティアを押し上げていることがわかる：

**$\beta = 0.5$（コスト重視）:**
- SC@64に対して**約69.5%のトークン削減**を達成しつつ、平均精度を維持
- これは「SC@64の全トークンの3割弱で同等の性能」を意味する

**$\beta = 1.0$（精度重視）:**
- 全ての手作りベースラインを超えるピーク精度を実現
- 8つの比較セルのうち5つで最高性能

![Token efficiency comparison across models](/images/autotts-agentic-test-time-scaling/fig4.png)

### 汎化性能

| 観点 | 結果 |
|------|------|
| 探索セット（AIME24） | CMCが全ベースラインを上回る |
| ホールドアウト（AIME25） | 精度・コストトレードオフでベースライン超え |
| ホールドアウト（HMMT25） | 同上 |
| モデルスケール汎化 | 4つのスケール（0.6B〜8B）のうち3つで平均精度一位 |
| 残り1スケール | Qwen3-8Bでも競争力を維持 |

AIME24で発見されたポリシーが、AIME25やHMMT25といった**未知のベンチマーク**に転移する点は特筆に値する。過学習の兆候は見られない。

### 探索コスト

| 項目 | 値 |
|------|---|
| 全体コスト | **$39.9** |
| 全体時間 | **160分** |
| 評価時LLM呼び出し | **0回** |

このコストにはClaude Codeの利用料が含まれる。オフラインリプレイ環境のおかげで、評価段階でのLLM呼び出しは完全にゼロだ。

## 考察

### 設計対象の転換の意義

AutoTTSの最大の貢献は、技術的な成果（CMCコントローラー）よりも**パラダイムの転換**にある。

従来：「人間が良いヒューリスティクスを考える」
AutoTTS：「良いヒューリスティクスが発見される環境を作る」

これはAI研究におけるメタレベルのシフトだ。TTSに限らず、プロンプトエンジニアリングやアーキテクチャ設計など、人間の直覚に依存している領域に応用可能な視点である。

### βパラメタライゼーションの elegancy

10次元のハイパーパラメータ空間を1次元のスイープに圧縮するという発想は、シンプルだが極めて効果的だ。制約を課すことで探索が安定化し、汎化性能が向上するというトレードオフは、機械学習全体の教訓（正則化）と共通している。

### CMCに見る「良い推論」の姿

CMCの設計から、LLMの推論効率化に何が重要かが浮かび上がる：

- **一時的な信頼度スパイクを無視する**（EMAトレンド要件）
- **合意形成中の枝に計算を集中させる**（アライメント認識）
- **多様性の維持**（最低2本の枝を常に保持）

これらは人間の集団意思決定にも通じる原理だ——過度な早期収束を防ぎ、意見の多様性を保ちつつ合意に向けて計算資源を配分する。

### 限界と課題

- **幅・深さのTTSに限定**: MCTSやSAT求解など、より一般的な探索パラダイムへの拡張は未検証
- **数学推論に限定**: AIME/HMMTでの評価のみで、コーディングや一般推論への転移は不明
- **事前収集軌跡への依存**: 軌跡品質がコントローラーの性能上限を規定する
- **5ラウンドという制約**: より多くのラウンドでどうなるかは未検証

## 関連研究との位置づけ

| 手法 | アプローチ | AutoTTSとの違い |
|------|-----------|----------------|
| SC@64 | 固定幅・多数決 | 適応的でない |
| ASC/ESC | 幅方向の適応 | 深さ方向を固定 |
| Parallel-Probe | 幅・深さの結合制御 | 手作りヒューリスティクス |
| R1 (DeepSeek) | 深さ方向のRL | 幅方向を未考慮 |
| AlphaOne | プロセス reward model | より高いコスト |
| Meta-search (lee2026meta) | メタ探索 | トレースフィードバックとの統合が異なる |

## まとめ

AutoTTSは、TTS戦略の設計を「手作り」から「環境駆動の自動発見」へと転換するフレームワークだ。オフラインリプレイ環境とβパラメタライゼーションの組み合わせにより、$39.9・160分という驚異的な低コストで高性能なコントローラーが発見される。

発見されたCMCコントローラーは、SC@64に対して**約69.5%のトークン削減**を達成しつつ精度を維持し、未知のベンチマーク・モデルスケールへの汎化も確認された。

この論文の真の価値は、特定のコントローラーの性能にあるのではなく、**「設計すべきは戦略ではなく環境である」という視点**を提示した点にある。このパラダイムは、TTSにとどまらずAIシステム設計全般に波及する可能性がある。
