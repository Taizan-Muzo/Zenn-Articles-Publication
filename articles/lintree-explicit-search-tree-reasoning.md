# 探索木を「見える化」する：親ポインタがLLM推論を劇的に改善する

> **LinTree: Improving LLM Reasoning with Explicitly Structured Search Histories**
> Liwei Kang, Yee Whye Teh, Wee Sun Lee (NUS / Oxford)
> arXiv:2605.31492 (2026年5月29日)

## TL;DR

- LLMの推論トレースは、裏側で「探索木」をなぞっているが、**木構造が暗黙的**すぎて活かしきれていない
- 探索履歴全体を条件付けできるはずのLLMが、局所状態だけを見るヒューリスティック探索にすら勝てない現象を発見
- **たった親ポインタ（parent pointer）を付与するだけで**、成功率・探索効率ともに劇的に改善
- Blocks World, Navigation, Sokobanの3環境でGRPO-explicitが**100%/100%/89.6%**を達成

## なぜこの論文が重要か

Chain-of-Thought (CoT) 以降、LLMの推論は「中間ステップの生成」で飛躍的に進歩した。思考の連鎖、自己修正、枝分かれ――モデルはまるで暗黙の探索木をたどるように解答を紡ぐ。

だが、ふと考えてほしい。**この探索木の構造をモデルは「見えている」のだろうか？**

CoTのテキストを「線形化された探索木」と捉える視点は既に存在する（Tree-of-Thought, LATSなど）。しかし、これらの多くはLLMの**外側**に外部コントローラを置き、LLM自身には現在パスの局所情報しか与えていない。

一方、単一のLLMが自前で推論トレースを生成する場合、モデルは**全探索履歴にアクセスできる**はずだ。局所ヒューリスティックより有利なはず――そう考えるのが自然だろう。

**本論文は、この「はず」を検証し、意外な結果とシンプルな解決策を提示した。**

![概念図：暗黙的 vs 明示的探索木](/images/lintree-explicit-search-tree-reasoning/fig1_concept.png)

## 問題設定：3つの制御可能な推論環境

本論文は3つの完全観測可能な探索タスクで評価する。

| 環境 | タスク | 難易度 |
|------|--------|--------|
| **Blocks World** | 積み木配置の初期→目標変形 | ★☆☆ |
| **Navigation** | 10×10グリッド上の障害物回避経路探索 | ★★☆ |
| **Sokoban** | 箱を押して目標位置に配置 | ★★★ |

各環境で20KのSFT訓練データを生成。探索トレースはルールベースヒューリスティック付きBest-First Searchで作成する。ベースモデルは**Qwen3-0.6B**、訓練は**H100 1枚**で完結する。

## 検証1：探索履歴へのアクセスは本当に有利か？

### トレース条件付け推論ポリシー

推論モデルは、以下のフォーマットで探索トレースを生成する：

```
EXPAND S{ 0:B<A<C ; 1:D }
EXPAND ACT C:A->D -> S{ 0:B<A ; 1:D<C }
GOAL_REACHED
```

SFTでBest-First Searchのトレースを学習し、その後GRPOで**効率ペナルティ付き報酬**を最適化する：

$$R(\tau) = \mathbf{1}[\text{valid} \land \text{correct}] \left(1 - \lambda \sum_{t=0}^{N_{\text{exp}}-1} \gamma^t \right)$$

$\lambda = 0.005$, $\gamma = 0.99$。正解かつ短いトレースほど高報酬。

### 局所状態LLMヒューリスティック

比較対象として、LLMをヒューリスティック関数として使うBest-First Searchも訓練。**現在の状態と目標だけ**を入力にして、各候補アクションの価値をスコアリングする。つまり「過去の探索履歴は見ない」。

### 結果：意外な敗北

| 方法 | Blocks World | Navigation | Sokoban |
|------|:---:|:---:|:---:|
| BFS (RL, local heuristic) | **99.8%** | **100.0%** | **99.1%** |
| GRPO (implicit trace) | 97.3% | 94.9% | 85.9% |

**暗黙的トレースを持つ推論モデルは、局所状態ヒューリスティックにすら勝てない。**

全探索履歴を見ているはずのモデルが、現在状態だけを見るヒューリスティックに負ける――このパラドックスの原因はどこにあるのか？

## 検証2：原因は「木構造の暗黙化」

著者らの洞察は鋭い。

> 探索トレースは線形化された探索木だが、**木のトポロジーがテキスト中に失われている**。
> モデルがバックトラックする際、「どのノードに戻ったか」が明示されていない。

「別の方法を試してみよう」といった自然言語の手がかりはあるが、**どの以前の探索状態を再訪しているのかが識別できない**。歴史は探索と回折の証拠を含んでいるが、木構造は文脈から推測するしかない。

そこで、最小限の変更を加えた**親ポインタ（parent pointer）**の導入を提案する。

## LinTree：親ポインタで探索木を明示化

### 変更は最小限

各EXPANDステップに、どのノードから拡張されたかを示す`sid`を付与するだけ：

```
# 暗黙的（従来）
EXPAND ACT C:A->D -> S{ 0:B<A ; 1:D<C }

# 明示的（LinTree）
EXPAND sid=0 ACT C:A->D -> sid=1 S{ 0:B<A ; 1:D<C }
```

SFTデータ・訓練ハイパーパラメータ・ベースモデルは**すべて同一**。差は親ポインタの有無だけだ。

### 結果：劇的改善

![成功率の比較](/images/lintree-explicit-search-tree-reasoning/fig2_success_rate.png)

| 方法 | Blocks World | Navigation | Sokoban |
|------|:---:|:---:|:---:|
| GRPO (Implicit) | 97.3% | 94.9% | 85.9% |
| GRPO (Explicit) | **100.0%** | **100.0%** | 89.6% |
| GRPO (Explicit + Constraints) | 100.0% | 100.0% | **98.9%** |
| BFS (RL, local heuristic) | 99.8% | 100.0% | 99.1% |

**SFT段階**だけで、暗黙的から明示的に変えるだけで平均5.0ポイントの改善。Navigationでは90.6%→95.2%、Sokobanでは74.8%→85.6%と劇的だ。

**GRPO後**にはBlocks WorldとNavigationで100%到達。Sokobanでも生成制約付きで98.9%に到達し、BFS（99.1%）にほぼ匹敵する。

### 探索効率も向上

![探索効率の比較](/images/lintree-explicit-search-tree-reasoning/fig3_efficiency.png)

親ポインタは成功率だけでなく、**必要な探索ステップ数も削減**する：

- Blocks World: 8.25 → **7.31**（-11%）
- Navigation: 14.80 → **14.28**（-4%）
- Sokoban: 63.54 → **52.82**（**-17%**）

SokobanではBFS（64.08回）よりも少ない拡張数で高い解決率を達成している。

## なぜ親ポインタが効くのか？

### 1. 計画抽出の信頼性向上

![計画抽出失敗率](/images/lintree-explicit-search-tree-reasoning/fig4_plan_extraction.png)

探索自体は正解プランを見つけているのに、最終的なアクションシーケンスを間違って出力するケースを「計画抽出失敗」と呼ぶ。

Sokobanでは暗黙的モデルが80.8%の失敗率に対し、明示的モデルは**54.2%**に低下。Blocks Worldでは41.4%→26.0%。

**木構造が見えていると、探索結果から正しいパスを抽出しやすくなる。**

### 2. 状態空間のより広範な探索

Navigation環境で、訪問状態の平均ペアワイズ距離を測定した：

| 方法 | $\bar{d}$（高ほど広範） |
|------|:---:|
| BFS (RL) | 3.99 |
| GRPO (Implicit) | 4.11 |
| GRPO (Explicit) | **4.19** |

明示的モデルは**より広く分散**して状態空間を探索する。親ポインタによって「既にこの辺りを探索した」という情報が可視化され、異なる方向への探索が促される。

### 3. SFTにおける構造模倣の改善

SFT段階でも顕著な改善（平均+5.0ポイント）があることから、親ポインタは**模倣学習そのものを容易にする**。木構造がテキスト中に埋め込まれていることで、モデルは探索の「振る舞いパターン」をより正確に復元できる。

## 重要な発見：BLOCKEDの記録も鍵

論文から見逃せないディテール。Navigation環境では、壁にぶつかった探索も`BLOCKED`としてトレースに**明示的に記録**している：

```
BLOCKED ACT L -> (5,5) WALL
```

これを省くと、SFT段階で暗黙的・明示的ともに有効な探索を学習できない。マンハッタン距離ヒューリスティックが強い方向バイアスを持つため、モデルは「最高スコアの候補が拡張される」という期待を持つ。壁で静かに枝刈りすると、可視的な拡張順序とヒューリスティック順位の不整合が生じて学習が阻害される。

## 実装の簡潔さが際立つ

この研究の美しさは、**提案手法のシンプルさ**にある。

- モデルの変更：**なし**
- 訓練パイプラインの変更：**なし**
- 追加コスト：**ほぼゼロ**（トークン数の微増のみ）
- 有効性：3環境すべてで一貫した改善

親ポインタの付与という、おそらく**最小の表現変更で最大の効果**を狙った設計は、今後の推論フォーマット設計に重要な示唆を与える。

## 関連研究との位置づけ

| 手法 | 検索制御 | LLMの入力 | 木構造 |
|------|---------|----------|--------|
| Tree-of-Thought | 外部コントローラ | 局所（現在思考） | 外部管理 |
| LATS (MCTS) | 外部MCTS | 現在パス | 外部管理 |
| LLM-A* | 外部A* | 局所（候補評価） | 外部管理 |
| **LinTree（本論文）** | **LLM自前** | **全探索履歴** | **トレース内に明示** |

Tree-of-ThoughtやLATSはLLMを「思考生成器」として使い、探索は外側のアルゴリズムが担う。対してLinTreeは、**LLM自身が探索と意思決定を統合**しつつ、親ポインタで探索の構造情報を保持するアプローチだ。

最近の`Re^2` (arXiv:2603.07197) は生産的でない推論パスを放棄して再開始する手法を提案しているが、LinTreeの親ポインタと相性が良さそうだ。「どのノードに戻るか」が明示されていれば、再開始の判断もより精度が上がるはず。

## 考察：推論の「インターフェース」を問い直す

本論文の最も深い含意は、手法そのものよりも**問い方**にある。

LLM推論の改善といえば、大抵は「もっと大きなモデル」「もっと良い訓練目的」「もっと多くのデータ」に話が向かう。だが本論文は別の軸を提示した：

> **推論トレースとその背後にある計算構造との「インターフェース」を改善せよ。**

親ポインタという最小のインターフェース改善で、Qwen3-0.6BがBlocks WorldとNavigationで100%に到達した。モデルの能力そのものを変えていないのに、表現の工夫だけでここまで変わる。

これは推論フォーマット設計（reasoning tokenの構造、CoTのフォーマット、思考の組織化）の重要性を力強く主張する結果だ。OpenAI o1やDeepSeek-R1のような推論特化モデルが内部でどのような構造情報を保持しているかはブラックボックスだが、LinTreeの示唆するのは、**「構造の可視化」自体が推論力を引き出すキーになりうる**ということだ。

## 限界と今後の課題

- **制御された環境に限定**：Blocks World等の完全観測可能タスクで検証。オープンドメイン（数学、コード）への一般化は今後の課題
- **1つのベースモデルのみ**：Qwen3-0.6Bで検証。大規模モデルでの効果は不明
- **外部転移関数の有無の公平性**：BFSは環境転移を正確に使えるが、LLM推論モデルは自由形式生成ゆえ誤りが起きる。生成制約で緩和できるが、完全な公平性は別の議論

## まとめ

LinTreeは、LLM推論を「探索」というレンズで捉え直し、**探索履歴の表現方法**という一見些細な問題から大きな発見を導いた。

- 探索履歴全体へのアクセスは**木構造が暗黙的だと活かしきれない**
- **親ポインタの付与だけで**成功率と効率が同時に改善
- 推論フォーマットの設計が、モデル能力の引き出し方を決める

「もっと賢いモデルを作る」だけでなく、「モデルの思考をどう構造化するか」。この視点は、推論モデルの次のブレイクスルーへの重要な一歩と言える。
