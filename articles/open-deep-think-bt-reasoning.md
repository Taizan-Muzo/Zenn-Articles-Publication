# OpenDeepThink: Bradley-Terry集約で並列推論の選択ボトルネックを解消

## TL;DR

- **並列サンプリングは簡単だが「どれが一番いいか」の選択がボトルネック**。Pointwise判定はノイズが多く、正解を見分けられない
- **OpenDeepThink**は、Bradley-Terryモデルによる**ペアワイズ比較→グローバルランキング**でこの壁を突破
- 比較時に生成される**自然言語フィードバックをそのまま突然変異に活用**し、集団ベースの進化的改善を実現
- Gemini 3.1 ProでCodeforces Elo **+405ポイント**（8ラウンド、約27分）。Deep Thinkの+411に匹敵
- **客観的検証可能なドメインでは有効**だが、主観的ドメインでは逆に性能低下。評価器の信頼性が成败を分ける

## 背景：テスト時計算スケーリングの次元

LLMの推論能力を引き出す手段として、**テスト時計算スケーリング（test-time compute scaling）**が急速に台頭している。o1やDeepSeek-R1に代表される現行の主流アプローチは、Chain-of-Thoughtを**深く**延ばすことで精度を稼ぐ――つまり「1本の道をできるだけ奥まで進む」戦略だ。

だが、このアプローチには構造的な限界がある。推論チェーンは本質的に**直列的**で、途中の1つのミスが最終結果を破壊する。長くすればするほど、そのリスクは増大する。

一方、**幅（breadth）のスケーリング**――複数の候補解を並列にサンプリングし、最良のものを選ぶBest-of-Nアプローチは、概念上は非常にシンプルだ。N個の解を生成して、一番よさそうなものを提出すればいい。

しかし、ここに**選択ボトルネック（selection bottleneck）**が立ちはだかる。

> どの候補が正解に近いかを判定するには、正解そのもの（ground-truth verifier）か、高精度な報酬モデルが必要だ。しかし現実には、どちらも手に入らないことが多い。

論文の実験ではこの問題を定量的に実証している。NOI-119の500ペアを使った診断テストでは：

- **ペアワイズ比較：86.2%の精度**
- **Pointwiseスコアリング：59.2%の精度**

Pointwise判定は**正解を認識するのは得意だが（AC recall 96.4%）、不正解を弾くのが壊滅的に下手（WA recall 62.2%）**。つまり「全員に高得点をつけてしまう」ポジティブバイアスが支配的で、差別化ができないのだ。

![ペアワイズ vs Pointwise精度比較](/images/open-deep-think-bt-reasoning/fig3.png)

このボトルネックこそが、並列推論の実用化を阻んできた最大の障壁だった。

## 方法：OpenDeepThinkフレームワーク

OpenDeepThinkは、この選択ボトルネックを**3つのアイデア**で解消する。

### 1. ペアワイズBradley-Terry比較

各候補を独立にスコアリングするのではなく、**ランダムなペアを組ませて比較させる**。これは人間の評価においてもpairwise comparisonのほうが信頼性が高いことが知られており、Chatbot Arenaも同じBradley-Terryモデルを使っている。

$$P(i \succ j) = \sigma(s_i - s_j)$$

ここで$s_i$は候補$i$のBTスコア、$\sigma$はロジスティックシグモイド関数。タイは各側に0.5勝としてカウントし、L-BFGSで正則化付き最尤推定を行う（$\lambda = 0.01$）。

**BTを使う意味**：各候補が直面する相手はランダムな部分集合に過ぎない。BTは**相手の強さを内部化して調整済みランキングを出力**するため、生の勝率では不可能な公正な順位付けができる。$K=4$という少ない比較数でも安定したランキングが得られる理由はここにある。

### 2. フィードバック駆動の突然変異

ここがOpenDeepThinkの最も独自的な部分だ。ペアワイズ比較の際に、判定だけでなく**自然言語のフィードバック（rationale）**を同時に出力させる。このフィードバックを**そのまま突然変異の入力**として再利用する。

```text
### Wins (this solution was judged better):
- "Efficient O(n log n) approach with correct sweep line"
### Losses (this solution was judged worse):
- "Misses the edge case where array length is 1"
```

突然変異プロンプトには3つの入力が与えられる：
1. 問題文
2. 現在の解法
3. 集約されたフィードバック

**重要な設計選択**：プロンプトはモデルに**「現在の解法を根本的に変えてもよい」**ことを許可している。これがないと、モデルは壊れた戦略にパッチを当て続けるだけで、構造的な見直しができない。

### 3. エリート保存と淘汰

各世代でBTスコアに基づいて：
- **上位25%**：エリートとしてそのまま保持
- **下位25%**：廃棄
- **上位75%**（エリート含む）：フィードバック付き突然変異

エリートは**保持と同時に突然変異も受ける**。これにより多様性を維持しつつ選択圧を維持する。

### 計算量設計

4つのハイパーパラメータ：個体数$n$、世代ごとの比較数$K$、世代数$T$、最終ラウンドの比較数$M$。

**メイン実験設定**：$n=20, K=4, T=3, M=10$

- **問題あたり約285回のAPIコール**
- **逐次深度はわずか8ラウンド**（各ラウンド内は並列実行）
- **壁時計時間：約27分**（Gemini 3.1 Proの場合）

この設計の肝は、**逐次深度が固定（8ラウンド）**であること。個体数$n$や比較数$K$を増やしても、ラウンド数は変わらない。並列度だけで精度をスケールできるのだ。

## 実験結果

### コンペティティブプログラミング：+405 Elo

CF-73（73問、Codeforces Div.1/2、レーティング2000-3100）とNOI-119（119問、国内情報オリンピック）の計192問で評価。

![世代ごとの精度変化（Hard Tier）](/images/open-deep-think-bt-reasoning/fig2.png)

最も注目すべきは**Hard問題**での結果だ。pass@1がわずか11%という超難問群で、3世代の進化を経てBT top-1が**50%に到達**。+39ポイントのゲインは：

- **+25：進化による候補解の質の向上**（サンプリングカバレッジの拡大）
- **+14：選択による精度の抽出**（BTランキングの精度）

Codeforces Eloに換算すると、Gen-0の2851から**3256へ+405ポイント**。これはGemini 3 Deep ThinkのLiveCodeBench Proでの+411に匹敵する。

> 「集約（aggregation）はベースモデルがほぼ解ける問題で支配的。進化（evolution）は能力のフロンティアで支配的。」

### クロスモデル転移

同一ハイパーパラメータ（調整なし）で3つのGeminiモデルで検証：

| モデル | Tier | gen-0 pass@1 | Final pass@1 | Final BT top-1 | Δ (pass / BT) |
|--------|------|-------------|-------------|---------------|---------------|
| 3.1 Pro | Hard (64問) | 11% | 36% | 50% | +25 / +27 |
| 3 Flash | Medium (75問) | 39% | 69% | 76% | +30 / +12 |
| 2.5 Pro | Easy (53問) | 28% | 50% | 60% | +22 / +18 |

パターンが非常にきれいだ：
- **弱いモデル（Flash）**：pass@1の伸びが大きい → 進化が主役
- **強いモデル（Pro）**：BT top-1の伸びが大きい → 選択が主役

モデル能力に応じて進化と選択の寄与がシフトする。いずれのモデルも、gen-0でどのモデルも解けなかった問題は最終的にも解けない――**OpenDeepThinkは部分的な能力を増幅するのであって、ゼロから新しい能力を生み出すわけではない**。

### クロスドメイン：HLEベンチマーク

HLE（Humanity's Last Exam）の82問で多分野に適用した結果が最も示唆に富む：

- **数学**：56% → 61%（+5pp）✅
- **生物学**：43% → 57%（+14pp）✅
- **物理学**：33% → 50%（+17pp）✅
- **人文・社会科学**：75% → 50%（**-25pp**）❌

**客観的に検証可能なドメインではBT選択が信号を増幅し、主観的なドメインではノイズを増幅する。**これは本質的に、ソフト検証器（LLM判定）が信頼できる領域とそうでない領域の境界線を描いている。

## 考察

### ペアワイズがPointwiseを圧倒する理由

対照実験がこれを決定的に示している。同一のgen-0プール（20候補）に300コール分の評価を投入した場合：

- **Pointwise単独**：Hardで26% top-1
- **Self-Refine + Pointwise**：Hardで41% top-1（生成品質はほぼ同じまで追いついた）
- **OpenDeepThink（BT）**：Hardで**50%** top-1

生成品質を一致させたうえでの**9ポイントの差は純粋に選択メカニズムの差**。Pointwiseは「全員に合格を出す」バイアスがあり、差別化ができない。一方ペアワイズは「AとB、どっちがマシか」という比較的容易な判別タスクに還元できる。

### ネガティブフィードバックが変異の全信号を担う

![フィードバック戦略アブレーション](/images/open-deep-think-bt-reasoning/fig4.png)

フィードバック戦略のアブレーションで最も驚かれるべき結果：

- **ポジティブフィードバックのみ**：ノーフィードバックと統計的に差なし
- **ネガティブフィードバックのみ**：ネット救助率が有意に向上
- **K=4が最適**、K=5で逆に低下

> 「何がよかったか」を教えるのは無意味。「何が悪かったか」を教えることで初めて、突然変異は意味のある修正を行える。

さらに、このフィードバックの効果は**Medium問題で顕著（+26pp）だが、Hard問題では縮小（+4pp）**。フィードバックは「受け入れの閾値の向こう側」へ押し上げることはできるが、**根本的に新しいアルゴリズムを思いつかせるわけではない**。

### 進化はフロントローデッド

世代ダイナミクスを見ると：
- **Gen 0→1で最大の改善**：修正可能な失敗の大部分を最初の突然変異で修正
- **Gen 1→2でスパース比較（K=4）が飽和**：候補が互いに同程度になると、少ない比較では順位を解決できない
- **最終密度ラウンド（M=10）でブレークスルー**：+8ポイントを追加抽出

スケーリング研究では、最適な個体数$n^*$が予算$B$に対して$n^* \propto B^{0.6}$で増加することが示されている。限界予算では**広いサンプリング>深い比較**が有利だ。

## 関連研究との位置づけ

| 手法 | 比較次元 | 選択信号 | 訓練不要 |
|------|---------|---------|---------|
| Self-Consistency | 多数決 | 最終解答の投票 | ✅ |
| Best-of-N + RM | 報酬モデル | 学習済みスコア | ❌ |
| Self-Refine | 自己評価 | 自己フィードバック | ✅ |
| FunSearch / AlphaEvolve | プログラム評価 | グラウンドトルース | ✅ |
| **OpenDeepThink** | **BTペアワイズ** | **LLM比較 + 進化** | **✅** |

OpenDeepThinkの独自性は：
1. **学習なし・検証器なし**で動作する
2. オープンエンド出力（コード全文など）にも適用可能
3. 選択と改善を**単一ループに統合**（比較→ランキング→突然変異→選択）
4. Chatbot Arenaの技術を**推論時の探索ループ内に持ち込んだ**初のケース

## まとめ

OpenDeepThinkは、並列推論の実用化を阻んでいた「選択ボトルネック」をBradley-Terryモデルと進化的フレームワークの組み合わせで解消した。Codeforcesで+405 Eloという圧倒的な結果は、**幅のスケーリングが深さのスケーリングに匹敵する潜在力を持つ**ことを示している。

同時に、HLEでの主観ドメインでの性能低下は**重要な注意喚起**だ。この手法は「ソフト検証器（LLMのペアワイズ判定）が信頼できる」という前提の上に成り立っており、その前提が崩れる領域では逆効果になる。

実用上の制約――約285コール/問題、約27分の壁時計時間――も無視できない。だが、強化学習を経ずに推論能力をここまで引き出せる手法は、**テスト時計算の新たなパラダイム**として間違いなく追跡に値する。
